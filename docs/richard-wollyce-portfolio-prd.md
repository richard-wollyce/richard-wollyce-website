# PRD — Richard Wollyce Portfolio Website

## Product overview

This document defines the portfolio website for Richard Wollyce. The site must function as a high-conviction professional asset for recruiters, hiring managers, founders, and technical leads evaluating him for software engineering roles.[cite:1]

The portfolio must communicate technical credibility, product ownership, business impact, and execution maturity within the first minute of interaction. It must feel clean, inspiring, and emotionally persuasive without becoming dramatic, decorative, or self-indulgent.[cite:1][cite:2]

## Product goal

The website exists to make a visitor reach a clear conclusion: Richard is a strong full-stack software engineer who can own important work from problem definition to production delivery.[cite:1]

The primary goal is to increase qualified outreach for interviews, engineering roles, and serious freelance opportunities. Every section must either establish trust, prove competence, or create a path to contact.[cite:1]

## Target audience

### Primary audience

- Recruiters hiring software engineers.[cite:1]
- Engineering managers hiring full-stack, product, or backend-leaning web engineers.[cite:1]
- Founders seeking a reliable engineer who can ship complete systems with limited supervision.[cite:1]

### Secondary audience

- Clients seeking freelance or contract software development.[cite:1]
- Teams interested in engineers with secure-by-default thinking, especially around access control, data protection, and infrastructure ownership.[cite:1]

## Positioning

### Core positioning statement

Richard Wollyce should be presented as a full-stack software engineer who builds secure, scalable web systems and takes real ownership across frontend, backend, database, and infrastructure.[cite:1]

### Perception target

After visiting the site, the user should feel that Richard is calm, technically sharp, trustworthy, modern, and already operating at the level of a dependable core team member.[cite:1][cite:2]

### Messaging rules

- Use one clear identity: Full Stack Software Engineer.[cite:1]
- Support that identity with proof of shipped systems, measurable outcomes, and technical range.[cite:1]
- Do not fragment the message into parallel labels such as creator, hacker, visionary, builder, or architect unless a specific context requires it.[cite:1]
- Do not use vague claims like “passionate,” “results-driven,” or “innovative” without evidence.[cite:1]

## Success criteria

The website is successful if a qualified visitor can understand who Richard is, what he builds, and why he is credible in 60 to 90 seconds.[cite:1]

### Primary conversion actions

- Visit LinkedIn.[cite:1]
- Open GitHub.[cite:1]
- Send an email.[cite:1]
- Download the CV.[cite:1]

### Evidence of success

- A recruiter can summarize Richard’s profile after scanning the hero and the first two sections.[cite:1]
- A hiring manager can identify business impact, technical ownership, and delivery quality from project content.[cite:1]
- Contact options are visible without forcing deep navigation.[cite:1]

## Information architecture

Only sections with strong hiring value should exist. Any section that does not add new, decision-relevant information must be removed.[cite:1]

### Required sections

1. Hero.[cite:1]
2. Selected Work.[cite:1]
3. Experience.[cite:1]
4. Technical Strength.[cite:1]
5. About.[cite:1]
6. Contact.[cite:1]

### Excluded sections

These sections should not exist unless substantive content is added later:[cite:1]

- Testimonials.[cite:1]
- Blog.[cite:1]
- Certifications.[cite:1]
- Generic services section.[cite:1]
- Skill bars or percentage indicators.[cite:1]
- Long personal manifesto.[cite:1]
- Timeline filler with weak milestones.[cite:1]
- Hobby or fun-facts section.[cite:1]

## Visual direction

### Design intent

The visual system must feel clean, precise, warm, and premium. It should communicate maturity and technical confidence rather than trend-chasing or cyberpunk theatrics.[cite:2]

### Palette direction

The attached portrait suggests a restrained palette built from charcoal, graphite, warm wood, soft stone, and a subtle amber or copper accent.[cite:2]

#### Recommended palette behavior

- Light mode background: warm off-white or soft stone.[cite:2]
- Dark mode background: charcoal or ink.[cite:2]
- Primary text: deep graphite.[cite:2]
- Accent: muted copper, bronze, or warm amber derived from the warm lighting and guitar tones in the portrait.[cite:2]
- Optional support accent: a restrained steel-blue note inspired by the glasses reflection, used sparingly only where it improves hierarchy.[cite:2]

### Visual rules

- Favor strong typography and layout over decoration.[cite:2]
- Use one main accent color, not a rainbow system.[cite:2]
- Keep most content left-aligned.[cite:2]
- Use the portrait as a trust anchor, not as a casual lifestyle collage.[cite:2]
- Avoid neon glows, abstract blobs, purple-blue gradients, or “hacker” clichés.[cite:2]

## Typography direction

Typography should carry much of the elegance and confidence of the site. A refined sans-serif for body and interface is the default safest direction, optionally paired with a subtle display face for major headings if used with restraint.[cite:3]

### Recommended font direction

- Body/UI: Satoshi, General Sans, or Inter.[cite:3]
- Display option for headings only: Boska, Zodiak, or Instrument Serif, used sparingly.[cite:3]

### Type rules

- Prioritize readability and hierarchy over visual flair.[cite:3]
- Avoid too many font styles or abrupt shifts in tone.[cite:3]
- Let weight, spacing, and scale create distinction before introducing decorative typography.[cite:3]

## UX requirements

### Experience principles

- The site must be fully responsive and mobile-first.[cite:3]
- Light and dark mode must both exist.[cite:3]
- The first screen must contain immediate clarity and a visible action path.[cite:3]
- Motion must be subtle, intentional, and non-blocking.[cite:3]
- Navigation must stay minimal.[cite:3]

### Navigation

Keep navigation short and purposeful:[cite:1]

- Work.[cite:1]
- Experience.[cite:1]
- About.[cite:1]
- Contact.[cite:1]

### Interaction rules

- Hover effects must apply only to interactive elements.[cite:3]
- No autoplay distractions.[cite:3]
- Project cards should feel crisp, readable, and clickable.[cite:3]
- The page should support fast scanning first and deeper reading second.[cite:3]

## Content and copy guidance

### Tone of voice

The copy must feel precise, calm, direct, and confident. It should sound like a capable engineer speaking clearly, not like a startup ad or a motivational page.[cite:1]

### Copy rules

- Every paragraph must do one job only: define, prove, or convert.[cite:1]
- Use compact, evidence-backed writing.[cite:1]
- Favor quantified proof over abstract descriptors.[cite:1]
- Remove repeated ideas even if they sound good in isolation.[cite:1]

### Required implementation features

- One H1 only.[cite:3]
- Accessible color contrast.[cite:3]
